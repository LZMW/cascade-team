**版本**: cascade-hybrid v3.0
**框架**: super-team-builder v3.0
**更新日期**: 2026-03-01
**团队类型**: 混合型（支持串行、并行、混合执行）

---

## 📋 团队概述

Cascade (级联战队) 是一个专业的软件开发团队，采用**6A框架**（Align-Architect-Atomize-Approve-Automate-Assess）进行结构化项目管理。团队采用**混合执行模式**，可根据任务特点智能选择串行、并行或混合执行方式。

### 核心特性

- ✅ **6A标准化流程**：从需求对齐到质量评估的完整开发流程
- ✅ **混合执行模式**：智能选择串行/并行/混合执行
- ✅ **质量门控**：每个阶段都有明确的质量检查点
- ✅ **MCP工具支持**：根据任务需要动态授权高级AI工具
- ✅ **完整文档产出**：每个阶段生成结构化文档

## 👥 团队成员

| 代号 | 角色 | 阶段 | 核心能力 |
|------|------|------|----------|
| **Anchor** | 需求对齐专家 | Align | 需求澄清、边界界定、验收标准 |

## 🔄 6A开发流程

```
┌──────────────────────────────────────────────────────────────────┐
│                      6A 项目开发工作流                            │
├──────────────────────────────────────────────────────────────────┤
│  1. Align    (对齐阶段)     模糊需求 → 精确规范       → Anchor    │
│  2. Architect (架构阶段)    共识文档 → 系统架构       → Atlas     │
│  3. Atomize  (原子化阶段)   架构设计 → 任务拆解       → Prism     │
│  4. Approve  (审批阶段)     原子任务 → 人工审查       → 用户审批  │
│  5. Automate (自动化执行)   按节点执行 → 代码实现      → Forge     │
│  6. Assess   (评估阶段)     执行结果 → 质量评估       → Scale     │
└──────────────────────────────────────────────────────────────────┘
```

### 执行模式

#### 🔗 串行模式（默认）
适用于完整项目开发，任务有强依赖关系：
```
Anchor → Atlas → Prism → [用户审批] → Forge → Scale
```

#### 🔀 并行模式
适用于相互独立的任务，可同时执行：
```
Forge-Module-A ∥ Forge-Module-B ∥ Forge-Module-C
```

#### 🔄 混合模式
适用于复杂的组合场景：
```
阶段1（串行）：Anchor → Atlas → Prism
阶段2（并行）：Forge-Module-A ∥ Forge-Module-B
阶段3（串行）：Scale 汇总评估
```

## 🚀 快速开始

### 1. 安装配置包

详见 [INSTALL.md](INSTALL.md)

### 2. 使用协调器

直接通过对话触发协调器：

```markdown
# 示例1：完整项目开发
"使用级联战队开发一个用户认证系统，需要登录、注册、密码重置功能"

# 示例2：单个阶段
"使用级联战队的Anchor帮我澄清这个需求"

# 示例3：并行开发
"使用级联战队同时开发三个独立的模块"
```

### 3. 查看执行过程

协调器会在项目目录创建 `.cascade/` 目录，包含所有阶段的产出：

```
{项目}/.cascade/
├── phases/                    # 串行阶段产出
│   ├── 01_align/
│   ├── 02_architect/
│   ├── 03_atomize/
│   ├── 04_automate/
│   └── 05_assess/
├── outputs/                   # 并行阶段产出
│   ├── forge-module-a/
│   ├── forge-module-b/
│   └── forge-module-c/
└── inbox.md                   # 消息中心
```

## 🎯 典型使用场景

### 场景1：新功能开发（串行模式）

```
用户请求 → 协调器分析
  → Anchor (需求对齐)
  → Atlas (架构设计)
  → Prism (任务拆解)
  → [用户审批]
  → Forge (代码实现)
  → Scale (质量评估)
  → 交付
```

### 场景2：多个独立模块（并行模式）

```
用户请求 → 协调器分析
  → 同时触发：
      ├─ Forge-Module-A
      ├─ Forge-Module-B
      └─ Forge-Module-C
  → Scale 汇总评估
  → 交付
```

### 场景3：复杂项目（混合模式）

```
用户请求 → 协调器分析
  → Anchor → Atlas → Prism (串行准备)
  → 同时触发 Forge-Module-A ∥ Forge-Module-B (并行开发)
  → Scale 汇总评估 (串行总结)
  → 交付
```

## 📊 质量门控

每个阶段完成后都有质量检查：

| 阶段 | 质量门控 | 检查项 |
|------|----------|--------|
| Align | ✓ 需求边界清晰 | 边界已确认、验收标准可测试、关键假设已确认 |
| Architect | ✓ 架构可行 | 架构图清晰、接口定义完整、与现有系统无冲突 |
| Atomize | ✓ 任务可执行 | 任务覆盖完整、依赖无循环、每个任务可独立验证 |
| Approve | ✓ 用户批准 | 执行检查清单通过、最终确认清单完成 |
| Automate | ✓ 代码可用 | 编译通过、测试通过、文档同步 |
| Assess | ✓ 交付完成 | 需求已实现、验收标准满足、交付物完整 |

## 📖 协作原则

1. **用户优先** - 不确定时主动询问，不要猜测
2. **灵活应变** - 模式是工具不是枷锁，根据实际情况调整
3. **结果导向** - 目标是完成任务，不是遵循流程
4. **透明沟通** - 向用户同步进度和决策
5. **质量门控** - 每阶段必须通过质量检查才能进入下一阶段
